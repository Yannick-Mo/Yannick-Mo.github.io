现代 C++ 线程同步：从互斥锁到原子操作

全面梳理 C++ 多线程同步机制，涵盖互斥锁、条件变量、信号量、原子操作、future/promise、latch/barrier、jthread 等，包含代码示例和最佳实践

一、什么是线程同步？

在多线程程序中，当多个线程同时访问共享数据时，可能出现数据竞争（data race），导致程序产生不可预知的结果。线程同步就是对多个线程的运行节奏进行协调，保证它们按照预期的顺序访问共享资源，避免数据的不一致性和程序的崩溃。

二、经典同步手段（C++11–C++17）

1. 互斥锁（std::mutex）与锁管理器

互斥锁是最基础的同步工具，确保同一时刻只有一个线程能访问被保护的临界区代码。

C++ 提供了多种互斥量类型：

类型 特点 适用场景
std::mutex 基本的互斥锁，不可递归 常规资源互斥访问
std::recursive_mutex 同一线程可重复加锁 递归函数内的加锁操作
std::shared_mutex 多线程共享读、写独占 读多写少的场景（如缓存）
std::atomic_flag 忙等待，不阻塞 短期锁，对延迟敏感的场景（自旋锁）

为了安全地管理锁，C++ 提供了 RAII 风格的锁管理器：

std::mutex mtx;

// lock_guard：构造加锁，析构解锁，简单高效
void safe_increment(int& val) {
    std::lock_guard<std::mutex> lock(mtx);
    ++val;
}

// unique_lock：更灵活，支持延迟加锁、提前解锁、与条件变量配合
void flexible_lock(int& val) {
    std::unique_lock<std::mutex> lock(mtx);  // 构造时加锁
    // ... 临界区操作
    lock.unlock();  // 提前解锁
    // ... 非临界区操作
    lock.lock();    // 再次加锁
}

🚨 重要：同一个 std::mutex 必须由同一个线程来进行 lock() 和 unlock() 操作，否则将产生未定义行为。因此务必使用 RAII 风格的锁管理器，以从根本上避免手动管理锁带来的风险。

2. 条件变量（std::condition_variable）

互斥锁只解决了"互斥访问"的问题，无法解决"线程间等待与通知"的需求。条件变量正是为这一场景而生的通信工具。

条件变量必须与 std::unique_lock 配合使用，核心方法：

  • wait()：线程释放锁并进入阻塞状态，直到收到通知
  • notify_one()：唤醒一个等待线程
  • notify_all()：唤醒所有等待线程

生产者-消费者模型：

std::mutex mtx;
std::condition_variable cv;
std::queue<int> data_queue;
constexpr int max_size = 10;

void producer() {
    for (int i = 0; i < 100; ++i) {
        std::unique_lock<std::mutex> lock(mtx);
        cv.wait(lock, []{ return data_queue.size() < max_size; });   // 队列不满才生产
        data_queue.push(i);
        lock.unlock();
        cv.notify_one();    // 通知消费者
    }
}

void consumer() {
    while (true) {
        std::unique_lock<std::mutex> lock(mtx);
        cv.wait(lock, []{ return !data_queue.empty(); });            // 队列非空才消费
        int val = data_queue.front();
        data_queue.pop();
        lock.unlock();
        cv.notify_one();    // 通知生产者
        // 处理 val ...
    }
}

⚠️ 条件变量容易出现虚假唤醒（spurious wakeup），因此必须在循环中检查等待条件，不能单凭 wait() 返回就假定条件已经满足。

3. 信号量（std::counting_semaphore，C++20）

信号量用于控制同时访问某资源的线程数量，即限制并发度：

  • std::counting_semaphore<Max>：可同时允许多个线程进入，并发上限由模板参数指定
  • std::binary_semaphore（等价于 counting_semaphore<1>）：作为更高效的互斥锁使用

#include <semaphore>
#include <thread>
#include <iostream>

std::counting_semaphore<3> sem(3);  // 最多 3 个线程同时访问

void worker(int id) {
    sem.acquire();   // 请求资源，可用数 -1
    std::cout << "Worker " << id << " is working..." << std::endl;
    std::this_thread::sleep_for(std::chrono::seconds(1));
    sem.release();   // 释放资源，可用数 +1
}

信号量比互斥锁更灵活（允许多个线程同时访问），比条件变量更轻量（不需要配合互斥锁）。

4. 异步操作与 Future/Promise

std::future 和 std::promise 提供了一种线程间传递计算结果的同步方式。

核心用法：

  • 生产者线程通过 std::promise::set_value() 将结果存入
  • 消费者线程通过 std::future::get() 阻塞等待结果

#include <future>
#include <thread>
#include <iostream>

void compute(std::promise<int> prom) {
    std::this_thread::sleep_for(std::chrono::seconds(1));
    prom.set_value(42);   // 存入结果
}

int main() {
    std::promise<int> prom;
    std::future<int> fut = prom.get_future();

    std::thread t(compute, std::move(prom));

    int result = fut.get();   // 阻塞等待结果
    std::cout << "Result: " << result << std::endl;
    t.join();
}

此外，std::async() 可以更方便地启动异步任务并获取 future 对象：

1
2
3
4

auto fut = std::async(std::launch::async, []{
    return 42;
});
std::cout << fut.get() << std::endl;   // 42

这种方式比手动管理锁和条件变量更加安全和清晰，适用于"一个线程产生结果、另一个线程消费结果"的场景。

5. 原子操作（std::atomic）

对于单个共享变量，使用互斥锁可能过于沉重。原子操作是更高效的替代方案，它保证对变量的访问不会被其他线程中断。

#include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> counter{0};

void increment() {
    for (int i = 0; i < 10000; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    std::thread t1(increment), t2(increment);
    t1.join();
    t2.join();
    std::cout << counter.load() << std::endl;   // 20000
}

主要类型：

  • std::atomic<T>：适用于任意可平凡复制的类型
  • std::atomic_flag：最小的原子布尔类型，保证无锁，通常用于实现自旋锁

内存序（memory order）：

内存序 用途 代价
memory_order_relaxed 计数器，不要求顺序 最小
memory_order_acquire 读端，与 release 配对 中等
memory_order_release 写端，与 acquire 配对 中等
memory_order_acq_rel read-modify-write 操作 中等
memory_order_seq_cst 默认，全局顺序一致 最大

原子操作适用于计数器、标志位、自旋锁和无锁数据结构等场景。对于简单场景，使用默认的 seq_cst 最安全。

三、现代 C++ 中的新同步手段（C++20 及以后）

1. std::latch（一次性屏障）

std::latch 是一种单次使用的向下计数器，线程可阻塞等待计数器归零。

特点：

  • 计数器只能递减，不能重置，归零后不可再用
  • 典型场景：主线程等待多个子线程完成初始化后再继续工作
  • count_down(n) 可一次性将计数器减少任意正整数

#include <latch>
#include <thread>
#include <iostream>

std::latch work_done(3);   // 需要 3 个线程完成

void worker() {
    std::this_thread::sleep_for(std::chrono::seconds(1));
    work_done.count_down();   // 计数器 -1
}

int main() {
    std::thread t1(worker), t2(worker), t3(worker);
    work_done.wait();         // 等待计数器归零
    std::cout << "All workers done!" << std::endl;
    t1.join(); t2.join(); t3.join();
}

2. std::barrier（可复用屏障）

与 latch 相比，std::barrier 在计数器归零后可以重置，用于多轮同步。

特点：

  • 可重复使用，每轮同步后自动复位
  • 支持在计数器归零时执行一个可选的"完成回调"
  • 适用于迭代算法中多个线程需要"步调一致"的场景

#include <barrier>

#include <iostream>

std::barrier sync_point(3, []{
    std::cout << "Phase complete" << std::endl;
});

void worker() {
    for (int i = 0; i < 3; ++i) {
        // 执行第 i 轮计算
        sync_point.arrive_and_wait();   // 等待所有线程到达
    }
}

3. 原子变量的 wait/notify 机制（C++20）

C++20 为 std::atomic 增加了类似条件变量的能力：

  • wait(val)：如果变量值不等于 val，立即返回；否则进入休眠等待
  • notify_one() / notify_all()：唤醒正在等待的线程

std::atomic<int> flag{0};

void waiter() {
    flag.wait(0);   // 等待 flag != 0
    // flag 已被修改，继续执行
}

void setter() {
    flag.store(1);
    flag.notify_one();   // 唤醒 waiter
}

性能优势： 由底层平台（如 Linux 的 futex）优化实现，相比 while (flag.load()) {} 这种方式更加省电且对 CPU 调度更友好。

4. std::osyncstream（同步输出流）

多线程程序中使用 std::cout 输出时，字符很容易被打散、交织在一起。std::osyncstream 通过 RAII 缓冲机制，确保一整个输出操作在析构时原子化地完成：

#include <syncstream>
#include <thread>
#include <iostream>

void worker(int id) {
    std::osyncstream(std::cout) << "Worker " << id << " starting" << std::endl;
    // 这一整行输出不会被其他线程的输出打断
}

5. std::jthread 与 std::stop_token（协作式线程）

std::jthread 是 std::thread 的现代化替代品，在析构时自动 join()，从根本上避免了因遗漏 join() 而导致的资源泄漏。

std::stop_token 提供了标准化的协作式停止机制：

#include <thread>
#include <stop_token>
#include <iostream>

void worker(std::stop_token stoken) {
    while (!stoken.stop_requested()) {
        // 执行工作...
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
    std::cout << "Worker stopped gracefully" << std::endl;
}

int main() {
    std::jthread jt(worker);    // 析构时自动 join
    std::this_thread::sleep_for(std::chrono::seconds(1));
    jt.request_stop();          // 请求停止
}   // jt 析构时自动 join

四、同步机制的对比与选择

同步手段 粒度 适用场景 注意事项
互斥锁 + RAII 临界区 保护任意长度的共享资源修改 注意锁粒度，避免死锁
条件变量 事件通知 生产者-消费者、线程等待某个条件满足 必须配合互斥锁，在循环中检查等待条件
信号量 资源计数 控制并发访问数量（如连接池） 比条件变量轻量，比互斥锁灵活
原子操作 单个变量 计数器、标志位、自旋锁 内存序选择需谨慎，默认用 seq_cst 最安全
future/promise 单次结果传递 异步任务的结果返回 适合"一次生产、一次消费"的场景
latch / barrier 多线程同步点 多阶段并行任务（一次性 / 可复用） C++20 特性，需要较新的编译器支持
原子 wait/notify 单变量事件等待 原子变量的条件等待 C++20 特性，比忙轮询省电友好

五、常见陷阱与最佳实践

死锁： 最常见的线程同步问题。当多个线程互相等待对方持有的资源时发生。避免死锁的策略包括：

  • 保持一致的加锁顺序
  • 使用 std::lock() 一次性锁定多个互斥锁
  • 使用 std::scoped_lock（C++17）自动处理多个锁的加锁顺序
1
2
3
4
5
6
7

std::mutex mtx1, mtx2;

// 使用 scoped_lock 一次锁定多个锁，避免死锁
void safe_transfer() {
    std::scoped_lock lock(mtx1, mtx2);   // C++17，自动处理加锁顺序
    // 同时持有 mtx1 和 mtx2
}

锁的粒度： 锁的范围过大，会导致线程串行化，失去多线程的性能优势；锁的范围过小，则可能遗漏共享资源的保护，引发数据竞争。

虚假唤醒： 使用条件变量时，wait() 返回不一定代表条件已经满足，必须在循环中检查等待条件，这是标准规范的要求。推荐使用带谓词的 wait 重载：

1
2
3
4
5
6

// 正确：带谓词的 wait，内部自动循环检查
cv.wait(lock, []{ return !queue.empty(); });

// 错误：直接 wait，容易被虚假唤醒欺骗
cv.wait(lock);
if (queue.empty()) return;   // 仍可能为空！

RAII 优先： 始终使用 std::lock_guard、std::unique_lock、std::scoped_lock 等 RAII 风格的锁管理器，避免手动调用 lock() 和 unlock()，从根源上消除因异常或分支导致忘记释放锁的风险。

// 正确：RAII 自动管理
void good() {
    std::lock_guard<std::mutex> lock(mtx);
    // 即使抛出异常，锁也会在析构时释放
}

// 错误：手动管理容易遗漏
void bad() {
    mtx.lock();
    // ... 如果此处抛出异常，锁永远不会释放
    mtx.unlock();
}
