1. MOC 是什么

MOC = Meta-Object Compiler（元对象编译器），是 Qt 在编译前的一个额外步骤。

你在 .h 文件里写 Q_OBJECT 宏，MOC 就分析这个头文件，生成一个 moc_ClassName.cpp。这个文件实现了 C++ 本身没有的反射能力——运行时知道类名、方法列表、参数类型等信息。

项目中 MOC 由 CMake 自动管理（CMakeLists.txt）：

1
2
3

set(CMAKE_AUTOMOC ON)   # 自动运行 moc
set(CMAKE_AUTOUIC ON)   # 自动编译 .ui 文件
set(CMAKE_AUTORCC ON)   # 自动编译 .qrc 资源

以 64 个 Q_OBJECT 类、70 个生成的 moc_*.cpp 文件为例（Qt 6.11.0），每个文件都是 MOC 的产物。

2. MOC 生成的五样东西

以一个典型类 WebSocketManager 为例，MOC 生成的文件（build/src/network/network_autogen/.../moc_WebSocketManager.cpp）包含：

生成物 作用
staticMetaObject 类的元信息（类名、父类链、方法列表、参数类型、属性、枚举等）
qt_static_metacall() 统一派发函数 — 通过 int index 调用任意 signal/slot/Q_INVOKABLE
qt_metacall() 虚函数，处理继承链上的方法调用（先调父类，再调自己的）
信号函数体 每个信号生成一个函数体，内容只有 QMetaObject::activate()
qt_metacast() 运行时类型转换（qobject_cast 的底层）
metaObject() 返回 &staticMetaObject（QML 场景下可能返回 dynamicMetaObject）

  • 访问权限（AccessPublic/AccessPrivate）
  • 类型标签（SignalData 表示这是一个信号，SlotData 表示这是一个槽）

4. 信号发射的完整链路

项目中，WebSocketManager::onTextMessageReceived 调用了 emit newMessageReceived(jsonObj)，下面是完整的执行链路：

第1步：emit 展开 → 调用信号函数

开发者写的：

emit newMessageReceived(jsonObj);

emit 是一个宏，展开为空。它真正的含义就是调用下面这个由 MOC 生成的普通成员函数。

MOC 生成的信号函数体（moc_WebSocketManager.cpp:259-262）：

// SIGNAL 7
void WebSocketManager::newMessageReceived(const QJsonObject & _t1)
{
    QMetaObject::activate<void>(this, &staticMetaObject, 7, nullptr, _t1);
}

信号就是一个有函数体的普通成员函数——包含参数且由 MOC 自动生成。

// 伪代码 — activate() 的逻辑
void QMetaObject::activate(QObject *sender, const QMetaObject *m, int local_signal_index, ...)
{
    QObjectPrivate::ConnectionData *cd = sender->d_func()->connectionData;
    QObjectPrivate::Connection *c = cd->connectionLists[local_signal_index].first;

    while (c) {
        if (c->receiver) {
            switch (c->connectionType) {
            case DirectConnection:
                // 同线程：直接调用
                c->receiver->qt_metacall(InvokeMetaMethod, c->method_index, args);
                break;
            case QueuedConnection:
                // 跨线程：包装成 QEvent，post 到事件循环
                QCoreApplication::postEvent(c->receiver, new QMetaCallEvent(...));
                break;
            case BlockingQueuedConnection:
                // 跨线程：postEvent + 阻塞等待完成
                // ...
                break;
            }
        }
        c = c->nextConnectionList;
    }
}

int WebSocketManager::qt_metacall(QMetaObject::Call _c, int _id, void **_a)
{
    // 先交给父类处理（父类可能也有自己的方法）
    _id = QObject::qt_metacall(_c, _id, _a);
    if (_id < 0)
        return _id;

    if (_c == QMetaObject::InvokeMetaMethod) {
        if (_id < 14)                           // 14 = 当前类的方法总数
            qt_static_metacall(this, _c, _id, _a);
        _id -= 14;                              // 减去自己的，剩余索引留给子类
    }
    // 处理 RegisterMethodArgumentMetaType 等
    return _id;
}

关键点：

  • _id 是扁平化索引：父类的方法在前，子类的方法在后
  • 先调父类的 qt_metacall() 消费父类的 _id 范围，再处理自己的
  • _id -= 14 返回剩余索引，让子类继续处理

_a 数组布局：

  • _a[0] = 返回值指针（void 类型时为 nullptr）
  • _a[1] 起 = 函数参数的指针数组

void WebSocketManager::qt_static_metacall(QObject *_o, QMetaObject::Call _c, int _id, void **_a)
{
    auto *_t = static_cast<WebSocketManager *>(_o);

    if (_c == QMetaObject::InvokeMetaMethod) {
        switch (_id) {
        case 0: _t->connected(); break;
        case 1: _t->disconnected(); break;
        case 2: _t->errorOccurred(
            (*reinterpret_cast<std::add_pointer_t<QString>>(_a[1]))
        ); break;
        case 3: _t->friendRequestReceived(
            (*reinterpret_cast<std::add_pointer_t<qint64>>(_a[1])),
            (*reinterpret_cast<std::add_pointer_t<qint64>>(_a[2])),
            (*reinterpret_cast<std::add_pointer_t<QString>>(_a[3]))
        ); break;
        // ...
        case 7: _t->newMessageReceived(
            (*reinterpret_cast<std::add_pointer_t<QJsonObject>>(_a[1]))
        ); break;
        case 8: _t->onConnected(); break;
        // ...
        case 13: _t->doReconnect(); break;
        }
    }

    // IndexOfMethod 分支：connect() 时通过函数指针查索引
    if (_c == QMetaObject::IndexOfMethod) {
        if (QtMocHelpers::indexOfMethod<void (...)>(
                _a, &WebSocketManager::newMessageReceived, 7))
            return;
        // ...
    }
}

InvokeMetaMethod 分支：通过 switch 按索引调用实际方法。参数从 _a[1] 开始解引用还原。

IndexOfMethod 分支：connect() 时通过函数指针找到对应的索引，完成"函数指针 → int index"的映射。

5. connect() 做了什么

调用 QObject::connect()：

1
2

connect(sender, &Sender::signal,
        receiver, &Receiver::slot);

内部步骤

  1. 匹配参数类型：确保信号和槽的参数类型兼容（信号参数可以比槽多，但必须前几个对齐）

  2. IndexOfMethod：通过 qt_static_metacall 的 IndexOfMethod 分支，把成员函数指针转为 signal_index 和 method_index（int 编号）

    1
2
3
4
5

    // connect() 内部伪代码
int signal_index = qt_static_metacall(sender, IndexOfMethod,
                                       0, reinterpret_cast<void**>(&signal_ptr));
int slot_index   = qt_static_metacall(receiver, IndexOfMethod,
                                       0, reinterpret_cast<void**>(&slot_ptr));

  3. 存入连接列表：在 sender 的 QObjectPrivate::ConnectionData 中插入一条 Connection

     1
 2
 3

    // 连接的数据结构（位于 Qt 私有实现中）
struct QObjectPrivate::Connection {
    QObject *receiver;          // 接收者对象指针
    QObject *sender;            // 发送者对象指针
    int signal_index;           // 信号索引
    int method_index;           // 槽/方法索引
    Qt::ConnectionType type;    // 连接类型
    bool isSingleShot;          // 单次触发
    // ...
};

C++ 没有反射，Qt 怎么做到的？

标准 C++ 没有运行时反射，所以 Qt：

  1. 编译时（MOC）：解析头文件，为每个类生成完整的元数据表
  2. 运行时（IndexOfMethod）：用成员函数模板 indexOfMethod<MethodType>() 匹配 staticMetaObject 中的方法表，比较函数指针找到索引

这相当于「在编译期提取类的结构信息，生成运行时可查询的表格」。

总结

概念 实质
Q_OBJECT 宏，告诉 MOC「这个类需要生成元对象代码」
moc 编译工具，解析头文件生成反射代码
staticMetaObject 类的元数据表（类名、方法列表、参数类型等）
信号函数体 由 MOC 生成，内部调用 QMetaObject::activate()
emit 宏，展开为空，实际就是调用信号函数
connect() 函数指针 → int 索引，存入连接列表
qt_static_metacall() switch 派发，int index → 实际函数调用
qt_metacall() 虚函数，处理父类→子类的索引链
activate() 查找连接列表，按连接类型分发调用
DirectConnection 直接调用 qt_metacall()，同线程同步
QueuedConnection postEvent() 到事件循环，跨线程异步

一句话概括：MOC 在编译期提取类的结构信息生成元数据表，emit 通过 activate() 查找连接列表，再用统一派发函数 qt_static_metacall() 通过 int 索引跳转到实际槽函数，从而绕过 C++ 缺乏反射的限制。
